## 三件套指标说明

- **orch_cost**: Thread 5 (orchestrator) 单 round 端到端 cycle 累加，每 round 1 条
- **sched_max**: 每 round 4 个 scheduler 线程 (Thread 0..3) sched_cost 取最大值；决定 round 实际延迟
- **wire_cost**: Thread 4 (wiring) 单 round 端到端 cycle 累加，每 round 1 条

## 关键观察 — orch 与 sched_max 反相关

5 次 run 在每个 run 内都出现一致模式：**sched_max 越小的 round 反而 orch_cost 越大、wire_cost 越小**。

- 所有 run 的 `Δorch (top10 − bot10)` 都是负值：top10 (sched_max 最大区间) 的 orch_cost 比 bot10 平均小 15~22us
- 所有 run 的 `Δwire (top10 − bot10)` 都是正值：top10 的 wire_cost 比 bot10 平均大 70~100us
- run 3 整体 orch_cost 最低 (trim15 mean 689.13us)，代价是 sched_max top10 最高 (1220.91us)

**解读**: 当 orch 跑得过快时会跟 wiring 线程争抢 SM 总线，把 wire_cost 拉高，进一步通过 sched 的 fanin/dispatch 链路把 sched_max 拉高。当前 orch ≈ 706us 已经接近 SM bus 争用的临界点 (参考 [`project_a5_p0_dead_cas_removal_regression`](../../../../.claude/projects/-home-pyptouser-chenpeng-simpler/memory/project_a5_p0_dead_cas_removal_regression.md) 记录的同一模式)。

## 噪声基线

- **wire_cost** 在 run 间最稳定 (极差 ~6us)
- **sched_max** 极差 ~22us，约 ±1.0% 量级
- **orch_cost** 极差 ~27us，约 ±2.0% 量级 (run 3 是离群点)

后续 perf 优化对比该基线时，建议:
- orch 变化 < 27us 视作噪声
- sched_max 涨 > 15us 即视为回归 (依据 [`feedback_perf_metric_beyond_sched_max_mean`](../../../../.claude/projects/-home-pyptouser-chenpeng-simpler/memory/feedback_perf_metric_beyond_sched_max_mean.md))
- 同时盯三件套，不能只看 orch_cost 单项

## 进一步优化方向

边际收益已经反转：继续压 orch 每减 1us 会换出 ~6~10us 的 sched_max。下一步应该寻找:
- 降低 SM bus 争用 (而不是继续压 orch hot path)
- 降低 wire latency 上限 (top10 wire 比 bot10 高 70~100us，尾部分布有改进空间)
